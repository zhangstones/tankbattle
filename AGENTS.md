# Agents Collaboration Rules

以下为必须遵守的项目协作要求：

1. 文档更新采用“按职责最小变更”原则（避免重复）：
- 添加功能特性、修改逻辑或修复 Bug 时，必须更新“受影响且唯一职责匹配”的文档，不要求所有文档同步重复改动。
- 文档侧重点：
- `README.md`：面向使用者，写“如何使用/如何运行/用户可见行为变化”。
- 需求文档（如 `REQUIREMENTS.md`）：写“目标、约束、验收标准”，不重复操作步骤。
- 方案设计文档：写“设计取舍、架构与实现方案”，不重复需求条目与用户手册。
- 常见问题（FAQ）：写“高频问题、排障与边界说明”，不重复完整功能说明。
- 若某次改动不影响某类文档职责范围，则不更新该文档；提交说明中注明“为何不需要更新”。
- 对应功能测试案例仍为强制更新（优先功能测试，必要时补充单元测试）。

2. 保持合理的模块化设计：
- 按职责拆分代码
- 控制模块耦合
- 提高可维护性与可测试性

3. 完成一个具体任务后，必须提交代码：
- 提交信息需清晰说明修改内容
- 备注本次任务范围与关键变更点

4. 审批与执行策略（接近全自动）：
- 默认尽量不询问，优先直接执行
- 修改代码无需单独批准，默认直接执行
- 仅在以下情况才询问：
- 平台权限强制要求审批
- 破坏性/高风险操作（如删除、重置、覆盖关键数据）

5. 依赖安装策略：
- 任务执行所需的 Python 包可自行安装
- 任务执行所需的 Golang 工具可自行安装

6. 构建验证策略：
- 修改代码后默认执行构建（至少 `go build ./...`）
- 如有 GUI 交付目标，默认同步构建可运行的 GUI 可执行文件
- 构建完成后默认重新运行程序，验证实际运行效果

7. 技能持续改进策略：
- 若发现技能（`skills/*/SKILL.md`）存在明显错误、歧义或失效步骤，需在当前任务中及时修正，不得拖延。
- 若在执行中学习到可复用的新方法、新回退路径或新排障经验，应及时优化对应技能并沉淀为可执行流程。
- 技能更新后应在提交说明中明确“修正点/新增经验点”，保证可追溯。

8. 需求合理性评审与异议机制（强制）：
- 接到需求后，必须先进行合理性分析，再进入实现。
- 若需求存在明显不合理（如与现有架构冲突、收益低且成本高、引入高回归风险、性能/安全隐患、与需求文档冲突），必须及时提出异议，不得直接硬做。
- 提出异议时必须同时给出可执行替代方案（至少 1 个），并说明取舍：实现成本、影响范围、风险、回退方式。
- 在用户未确认高风险方案前，不得推进高风险实现；可先落地低风险部分以减少返工。
- 若最终按用户坚持方案执行，需在提交说明中明确“已提示风险 + 用户决策 + 兜底措施”。
