# Agents Collaboration Rules

本文件定义本项目的默认协作约束。目标是让执行方式稳定、规则边界清晰，并方便后续继续追加新规则而不造成重复或冲突。

## 1. 使用原则

### 1.1 适用范围
- 本文件适用于本仓库内的所有代理协作、代码修改、文档更新、验证与提交流程。
- 若下级目录存在更具体的 `AGENTS.md`，则更具体的规则优先。

### 1.2 必要文档索引
- 开始任务前，默认优先查看以下必要文档，而不是遍历全部 Markdown 文档。
- 默认优先查阅：`README.md`、`REQUIREMENTS.md`、`DESIGN.md`、`BUILD.md`、`FAQ.md`。
- 角色分工与职责边界见 `ROLES.md`；仅在任务涉及协作分工、角色边界或交接方式时优先查阅。
- 各文档职责定义见“3.1 文档更新规则”中的“文档职责边界”。
- `PLAN.md`、`ISSUE.md`、测试专项文档等仅在任务明确相关时再查阅，不进入默认必要索引。

### 1.3 总体执行原则
- 默认直接执行，尽量减少不必要询问。
- 优先做低风险、可验证、可回退的改动。
- 保持模块化、低耦合、可维护、可测试。
- 文档按职责最小变更，避免多处重复维护同一信息。

### 1.4 新规则追加方式
- 新规则应添加到“职责最匹配”的章节，不重复写到多个章节。
- 每条新规则尽量写清楚 4 个要素：`适用场景`、`默认动作`、`例外条件`、`落地产物`。
- 若新规则会覆盖旧规则，必须显式写明优先级或替代关系，避免并存冲突。

## 2. 标准工作流

### 2.1 先遵守 PDCA 原则
- 无论执行者是什么角色，在开始任务前都应遵守 PDCA 原则。
- PDCA 在本项目中作为经验法则（rule of thumb）使用，用于约束基本工作节奏，而不是强制引入冗长流程。
- `Plan`：先明确目标、边界、风险与落地产物。
- `Do`：按当前任务范围执行必要修改，避免无关扩散。
- `Check`：通过构建、运行、测试、审阅等方式确认结果符合预期。
- `Act`：根据检查结果修正问题，并沉淀可复用经验到代码、文档、测试或技能中。

### 2.2 先做合理性评审
- 接到需求后，必须先判断需求是否合理，再进入实现。
- 若需求明显不合理，不得直接硬做，必须先提出异议并给出替代方案。

### 2.3 再做实现
- 默认无需单独批准，直接完成必要修改。
- 实现时优先控制影响范围，避免顺手做无关重构。

### 2.4 完成后做验证
- 代码变更后，默认执行构建验证，至少运行 `go build ./...`。
- 若交付目标包含 GUI，默认同步构建可运行的 GUI 可执行文件，并重新运行程序验证实际效果。
- 若本次仅修改文档或纯说明性内容，可不执行构建，但需在提交说明中注明原因。

### 2.5 最后提交
- 完成一个具体任务后，必须提交代码。
- 提交信息需清晰说明修改内容、任务范围与关键变更点。

## 3. 专项规则

### 3.1 文档更新规则

#### 默认原则
- 文档更新采用“按职责最小变更”原则，避免重复。
- 添加功能特性、修改逻辑或修复 Bug 时，必须更新“受影响且唯一职责匹配”的文档。
- 不要求所有文档同步重复改动同一信息。

#### 文档职责边界
- `README.md`：面向使用者，记录如何使用、如何运行、以及用户可见行为变化。
- 需求文档（如 `REQUIREMENTS.md`）：记录目标、约束、验收标准，不重复操作步骤。
- 方案设计文档：记录设计取舍、架构与实现方案，不重复需求条目与用户手册。
- FAQ 文档：记录高频问题、排障方法与边界说明，不重复完整功能说明。

#### 落地要求
- 若某次改动不影响某类文档职责范围，则不更新该文档。
- 提交说明中需注明“为何不需要更新该类文档”。
- 对应功能测试案例仍为强制更新；优先补功能测试，必要时补充单元测试。

### 3.2 模块化设计规则
- 按职责拆分代码，避免单个模块承担过多责任。
- 控制模块耦合，避免隐式依赖和跨层污染。
- 优先提高可维护性与可测试性，而不是追求短期省事。

### 3.3 分层、模块化与信息力度控制规则
- 管理文档、代码、测试案例时，必须同时考虑分层、模块化设计与信息力度控制。
- 应按职责与受众拆分内容，避免把不同层级、不同目标、不同抽象度的信息混杂在同一处。
- 信息表达应控制在当前任务所需的合适粒度，避免一次性暴露过多细节，增加认知负担。
- 应优先采用渐进披露方式组织内容：先给必要信息，再在需要时展开实现细节、边界说明和专项内容。
- 若发现文档、代码或测试案例已经违反上述原则，应及时重构和优化，而不是继续在原有混乱结构上叠加修改。

### 3.4 并行协作边界规则
- 整个项目默认处于多人并行协作状态，执行任务时必须假设其他人也可能正在修改仓库中的其他部分。
- 项目协作角色固定包含：`Planner`、`Developer`、`Tester`、`Reviewer`；默认按多人并行协作处理。
- 默认只修改与当前任务直接相关的文档、代码和测试，不相关的改动不要去动。
- 不得为了顺手整理或个人偏好，修改与当前任务无关的结构、命名、格式或行为。
- 若发现他人的进行中改动、历史遗留问题或无关异常，但与当前任务不直接冲突，应保持不动，并在必要时单独说明。
- 若无关改动已经影响当前任务推进，应先收敛影响范围，必要时明确说明冲突点后再处理。

### 3.5 角色与共享文档协作规则
- 整个项目的协作默认通过共享代码仓库以及共享文档完成，重点协作文档包括 `docs/PLAN.md` 与 `docs/ISSUE.md`。
- 稳定的角色分工、职责边界与交接方式统一定义在 `ROLES.md`，`AGENTS.md` 不重复展开整套角色说明。
- `Planner`、`Developer`、`Tester`、`Reviewer` 在协作时应优先通过文档同步上下文、分工、阻塞项与结论，而不是依赖口头假设。
- 在 `docs/PLAN.md`、`docs/ISSUE.md` 或其他承担协作职责的文档中新增事项时，必须尽量备注清楚：`提出角色`、`负责角色`。
- 若当前执行者无法明确自己的协作角色，允许直接询问用户，不得在角色不清晰的情况下擅自篡改他人职责边界。

### 3.6 状态跟踪规则
- 需求、任务和代办事项的状态，必须在相关文档中持续跟踪，而不是只保留在临时上下文里。
- 状态跟踪默认使用 checkbox 形式维护，例如 `- [ ]`、`- [x]`，必要时再补充简短状态说明。
- `docs/PLAN.md` 侧重记录计划内任务、分工和推进状态；`docs/ISSUE.md` 侧重记录问题、风险、阻塞与处理状态。
- 当任务状态发生变化时，应同步更新对应文档中的 checkbox，避免代码状态与文档状态脱节。

### 3.7 审批与执行规则

#### 默认策略
- 默认尽量不询问，优先直接执行。
- 修改代码、文档、测试时，无需单独申请批准。

#### 必须询问的情况
- 平台权限强制要求审批。
- 破坏性或高风险操作，例如删除、重置、覆盖关键数据。

### 3.8 依赖安装规则
- 任务执行所需的 Python 包，可自行安装。
- 任务执行所需的 Golang 工具，可自行安装。
- 安装行为应以完成当前任务为目标，避免引入无关依赖。

### 3.9 技能持续改进规则
- 若发现技能文件（`skills/*/SKILL.md`）存在明显错误、歧义或失效步骤，必须在当前任务中及时修正，不得拖延。
- 若在执行中学到可复用的新方法、新回退路径或新排障经验，应同步沉淀到对应技能中，形成可执行流程。
- 技能更新后，应在提交说明中明确写出“修正点”或“新增经验点”，保证可追溯。

### 3.10 需求异议机制

#### 必须提出异议的情况
- 与现有架构明显冲突。
- 收益低但实现或维护成本明显过高。
- 引入高回归风险。
- 存在性能或安全隐患。
- 与现有需求文档冲突。

#### 提出异议时的要求
- 必须同时给出至少 1 个可执行替代方案。
- 必须说明取舍：实现成本、影响范围、风险、回退方式。

#### 执行边界
- 在用户未确认高风险方案前，不得推进高风险实现。
- 可先落地低风险部分，以减少返工。
- 若最终按用户坚持方案执行，提交说明中必须明确记录：已提示风险、用户决策、兜底措施。

## 4. 提交说明要求
- 提交说明应覆盖本次任务范围。
- 提交说明应概括关键变更点。
- 若跳过构建、运行、某类文档更新或某项验证，必须写明原因。
- 若包含技能修正或高风险方案取舍，必须写明对应结论。

## 5. 维护目标
- 本文件应优先保持“可执行”而非“口号式”表述。
- 新增规则时，优先补充边界与例外，避免只写原则不写执行方式。
- 若后续规则明显增多，应继续按职责拆分章节，而不是在原章节下无限追加混合条目。
