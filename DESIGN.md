# TankBattle Design

本文档说明当前代码实现的设计边界、模块职责与关键取舍，重点回答“为什么这样设计”以及“后续如何在不破坏结构的前提下继续演进”。

## 1. 设计目标与边界

- 目标：在 Ebiten 单线程主循环内，保持玩法逻辑稳定、状态切换清晰、表现层可持续迭代。
- 边界：
  - 需求、约束和验收标准以 `REQUIREMENTS.md` 为准。
  - 运行方式、操作说明和用户可见行为以 `README.md` 为准。
  - 本文档聚焦模块职责、状态机边界、渲染分层和可维护性策略。

## 2. 总体结构

代码采用“单个 `game` 聚合对象 + 按职责拆分文件”的结构：

- `model.go`
  - 定义常量、实体结构、状态枚举与 `game` 聚合根。
- `setup.go`
  - 负责新对局初始化、地图构建、波次启动、敌人生成。
- `state.go`
  - 负责输入分发、主状态机切换、子系统更新时序。
- `menu.go`
  - 负责菜单输入、配置变更与“恢复原对局 / 强制重开”的策略。
- `movement.go`
  - 负责玩家移动、转向，以及敌人 AI 移动与寻路决策。
- `combat.go`
  - 负责子弹推进、碰撞与伤害结算。
- `powerup.go`
  - 负责道具刷新、拾取、生效与过期。
- `render.go`
  - 负责场景、HUD、菜单、状态弹层、历史面板的具体绘制编排。
- `render_theme.go`
  - 负责统一 UI 调色板、面板/描边/发光/进度条等通用绘制原语。
- `debug_api.go`
  - 负责调试控制器、本地 HTTP 调试接口、主循环内动作执行与界面快照导出。
- `settings.go` + `score_history.go`
  - 负责用户配置与历史战绩的持久化。
- `audio_manager.go`
  - 负责音效播放、冷却、优先级与配置同步。

## 3. 状态机设计

主状态：

- `stateMenu`
- `statePlaying`
- `stateEnded`

关键流转：

- 菜单开始游戏：`menuStart` -> `startMatch()` -> `statePlaying`
- 对局结束：`finishMatch(win)` -> `stateEnded`
- 对局中进入菜单：`enterMenuForConfig()` -> `stateMenu`
- 菜单返回对局：`leaveMenuByToggle()`
  - 仅改音频配置：恢复原对局状态
  - 改难度或总波数：重开新对局应用配置

关键设计点：

- `game` 保存 `menuReturnState`、`menuReturnPaused`、`menuRequireRestart`，把“菜单入口”和“恢复策略”集中管理，避免 UI 层与玩法层相互耦合。

## 4. 更新时序

`Update()` 中的主顺序：

1. 全局热键与历史面板切换
2. 重开与菜单切换
3. 暂停判断
4. 玩家更新
5. 敌人更新
6. 子弹更新
7. 道具更新
8. 爆炸与墙体清理
9. 胜负判定与波次推进

该顺序保证：

- 输入优先，行为可预测
- 同一帧内完成主要战斗结算
- 波次与胜负结论基于最新状态作出

## 5. 表现层分层策略

当前界面重设计后，渲染层采用“主题原语 + 具体场景编排”的两层结构：

- 主题原语层：`render_theme.go`
  - 统一颜色、发光、描边、面板、进度条、选中态等低层表现能力。
- 场景编排层：`render.go`
  - 组合主题原语，绘制菜单、HUD、状态弹层、历史面板与战场对象。

这样拆分的目的：

- 菜单、HUD、历史面板共享同一视觉语言，避免重复堆叠颜色和边框逻辑。
- 表现层修改尽量局限在渲染文件，不影响状态机、AI、战斗与持久化逻辑。
- 后续如果要继续增强 UI，只需要扩展主题原语或调整局部编排，不必重新改写核心玩法代码。

## 6. UI 分层与遮挡原则

渲染顺序遵循：

1. 背景层
2. 地图与实体层
3. 战斗特效层
4. HUD 层
5. 历史面板层
6. 消息 / 暂停 / 结算弹层

关键原则：

- HUD 必须避开消息提示框，避免信息互相遮挡。
- 历史面板显示时优先级高于普通提示，防止被中途状态框打断阅读。
- 战场特效增强后，主体辨识度仍优先于装饰性效果。

## 7. 调试 API 与可测试性边界

为了支持功能测试，当前增加了一套“主循环内处理”的调试 API：

- 入口由 `RunWithOptions` / `TANKBATTLE_DEBUG_API_ADDR` 启用。
- 对外通过本地 HTTP 接口接收动作与快照请求。
- 对内通过 `DebugController` 把请求投递到游戏主循环，由 `Update()` 消费。

这样设计的原因：

- `ebiten.Image.ReadPixels` 必须在主循环启动后使用，不能依赖循环外的随手截图。
- 菜单操作直接复用现有状态机和 `applyMenuAction`，避免测试路径和真实逻辑分叉。
- 快照由游戏自己导出，避免外部桌面截图带来的焦点、遮挡、窗口层级干扰。

调试模式的约束：

- 使用固定随机种子，保持地图、敌人刷新等结果可重复。
- 不读取或写入用户本地配置与历史数据，避免测试污染用户环境。
- 功能测试若需要导出快照，应优先使用调试 API，而不是操作系统层截图。

## 8. 可测试性策略

- 布局相关规则通过 `render_test.go` 维护，例如：
  - 菜单标题居中
  - 菜单选项与底部提示区不重叠
  - HUD 与历史面板尺寸保持在屏幕范围内
  - 历史面板行距保持可读
- 调试能力通过 `debug_api_test.go` 维护，例如：
  - 调试动作执行
  - 调试状态查询
  - 固定随机种子下的确定性初始化
  - 快照导出路径校验
- 行为逻辑仍优先通过功能测试覆盖状态切换、菜单配置、战斗结算、AI 与数据持久化。

## 9. 后续演进规则

出现以下变化时，应同步更新本文档：

- 渲染层再次拆分模块或新增新的表现层子系统
- 状态机新增主状态或改变关键流转
- UI 分层/遮挡优先级发生变化
- 持久化结构或路径发生变化
