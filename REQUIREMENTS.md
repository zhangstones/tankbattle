# 坦克大战需求规格说明

## 1. 项目目标
使用 Go 开发 Windows 平台坦克大战游戏，核心为堡垒防守玩法，支持难度配置、清晰界面和可靠测试覆盖。

## 2. 平台与运行环境
- 操作系统：Windows
- 开发语言：Go
- 游戏引擎：Ebiten
- 运行形态：
  - GUI 可执行文件需无命令行黑窗（`windowsgui` 构建方式）

## 3. 核心玩法需求
- 玩家操作：
  - 平移移动：`WASD` / 方向键长按
  - 转向：`WASD` / 方向键双击同方向触发
  - 开火：`J` / `Space`
  - 需支持平移射击（移动过程不强制改车体朝向，转向由双击触发）
- 对战目标：
  - 保护底部堡垒。
  - 清空所有波次且堡垒存活为胜利。
  - 玩家 HP 归零或堡垒 HP 归零为失败。
- 重开行为：
  - `R` 可在运行中即时重开（不限于结算后）。
- 暂停与菜单行为：
  - `P` 在战斗中暂停/继续。
  - `P` 触发暂停时，界面仅显示单一暂停提示框，不得叠加额外状态提示框。
  - `M` 可从战斗中或结算界面进入菜单；在菜单中再次按 `M` 应可返回上一状态。
  - 若仅修改音效开关或音量，`M` 返回应恢复原对局（包括恢复到暂停态）。
  - 若修改难度或总波次，`M` 返回应自动重开对局并应用新参数。

## 4. 敌人与 AI 需求
- 每波敌人数：
  - 支持基础数量配置，并按波次增长，且有上下限。
  - 该参数由难度策略自动控制，不在菜单中单独提供配置项。
- 总波次：
  - 需支持总波数独立配置，范围 `1~5`。
  - 难度切换时可应用推荐总波数，但允许手动覆盖。
- 刷新方向分布：
  - 不能只从单一方向刷新。
  - 刷新点应覆盖上/左/右三侧。
- 重叠约束：
  - 必须避免敌人与敌人、敌人与玩家重叠。
- 移动与寻路：
  - 使用局部实时流场风格转向。
  - 包含防卡死逻辑，避免坦克原地冻结。
  - 抑制炮管/方向快速反复横跳（转向平滑/锁定）。
  - 每个敌方坦克必须具备独立随机行为因子（重规划节奏、开火倾向、进攻性）。
  - 随着对局时间推进，敌方目标偏好应逐步收敛到堡垒。
  - 当玩家长时间不开火时，敌方命中堡垒概率需持续上升（主动加压），不得直接脚本扣堡垒 HP。
  - 加压阶段不得使用斜向定点炮弹；仅允许通过路径/朝向调整来提高正向命中堡垒概率。
  - 即使只剩 1 辆敌方坦克，在玩家长期不开火时，也必须最终通过攻击行为破坏堡垒。

## 5. 碰撞与战斗需求
- 碰撞覆盖：
  - 坦克与墙/堡垒/边界/坦克。
  - 子弹与墙/堡垒/玩家/敌人/边界。
- 子弹行为：
  - 区分玩家与敌方子弹归属。
  - 敌我子弹发生碰撞时，双方子弹需互相抵消并销毁。
  - 按目标类型正确结算伤害。
  - 敌方命中护墙扣分：`-2`；命中堡垒扣分：`-3`。
  - 敌方每次命中堡垒，固定造成 `3` 点堡垒完整性伤害。
  - 开火冷却需可配置且节奏稳定：普通开火、连发开火与敌方开火应有明确区分，避免过快刷弹或过慢失去压迫感。
- 墙体：
  - 支持可破坏与不可破坏两类墙。
  - 护墙命中应按正方形分块渐进损毁，不允许一次性全毁，也不允许每次仅极小不可感知损耗。
  - 堡垒护墙每个分块需被命中 2 次后才销毁。
  - 可破坏障碍墙同样采用分块损毁：每次命中销毁一小块，不整段同时消失。
  - 可破坏墙分块尺寸应尽量与坦克宽度一致，确保单次命中后形成可通行缺口。
- 堡垒：
  - 具备 HP 与可视化完整度条。
  - 需对齐网格线，且距离屏幕底边固定 1 个网格。

## 6. 道具需求
- 道具类型：
  - 护盾
  - 连发
  - 堡垒修复
- 道具来源：
  - 地图随机刷新
  - 敌人概率掉落
- 生命周期：
  - 存在时间到期消失
  - 拾取后生效，必要时带上限/封顶控制

## 7. UI/UX 需求
- 菜单可用性：
  - 菜单可直接操作且易理解。
  - 菜单整体布局需保持足够留白，顶部说明区与菜单选项区不得出现拥挤或遮挡。
  - 菜单说明区与选项区需保持紧凑可读间距，并可随可用空间进行自适应计算，避免两区域距离异常。
  - 菜单标题与说明文本需保持视觉居中对齐，不得出现明显偏移。
  - 菜单标题需在标题栏内上下居中；关键操作提示（如 `FIRE J/Space`）需与 Combat 说明同组展示，保持整体阅读连续性。
  - 菜单界面按 `Enter` 可直接开始游戏。
  - 支持难度与总波数配置。
  - 提供快捷键提示（`1/2/3` 等）。
  - 菜单项需有可读文字说明和影响描述。
- HUD/状态栏：
  - 显示 HP、得分、历史最高分、当前排名、波次、敌人数、Buff、堡垒完整度。
  - 历史分数明细不直接显示在 HUD 上，应通过独立历史面板查看（如 `H` 键开关）。
  - `H` 键切换历史面板时，不应弹出额外提示框遮挡战斗视图。
  - 历史面板需支持滚动查看排名记录，并支持至少 100 条历史数据。
  - 历史面板的表格宽度与行高需保持充足留白，不得出现明显拥挤。
  - 右侧仅保留两条能量条：堡垒完整性、坦克总完整性（车体+炮塔合并）。
  - 坦克总完整性默认 10（车体 5 + 炮塔 5），数值变化应与中弹损伤保持可解释的一致关系。
  - 失败态下，仅失败方对应能量条归零并使用醒目告警色标记。
  - 状态栏背景宽度需随内容自适应，避免固定过宽。
  - 屏幕宽高需保持网格尺寸的整数倍，避免网格与对象错位。
  - 波次/状态提示框与 HUD 不得重叠，需自动避让。
  - HUD 与状态提示框边框应保持细线风格（建议 1-2px inset），避免边框过粗影响信息可读性。
- 音效与分发：
  - 第一阶段需提供写实风格音效（至少覆盖射击、命中、爆炸、道具、菜单/暂停、胜负结算）。
  - 需覆盖波次状态提示音（准备阶段、开波提示）。
  - 需覆盖菜单边界操作提示音（参数达上下限时）。
  - 道具拾取音需支持按类型区分（护盾/连发/修复）。
  - 需覆盖 Buff 结束音（护盾结束、连发结束）与敌我摧毁区分音。
  - 音效冷却计时需独立于战斗逻辑帧计数，菜单状态与重开后仍应稳定触发音效。
  - 菜单中需提供音效总开关（`Sound Effects` On/Off）。
  - 菜单中需提供音效音量档位调节（0~100%，可离散步进）。
  - 音效配置（开关、音量）需支持本地持久化并在启动时恢复。
  - 本地持久化配置文件不得写入当前工作目录，需写入用户目录下的 `~/.tankbattle/`。
  - 混音需支持基础分组（如 combat/ui/state/powerup）与状态优先级，关键状态音触发时可临时压低其他组。
  - 音效资源需通过 Go `embed` 打包到二进制中，保证分发时无需额外资源目录。
- 视觉风格：
  - 坦克与墙体应更真实/风格化，不是纯色方块。

## 8. 构建与启动需求
- 同时提供：
  - 开发运行路径（`go run ...`）
  - 用户游玩 GUI 可执行文件路径（无黑窗）
- 推荐游玩二进制：
  - `tankbattle_gui.exe`

## 9. 测试需求
- 必须同时包含单元测试与功能测试。
- 测试总量需 >= 30。
- 覆盖主要逻辑路径，包括：
  - 菜单动作与状态切换
  - 难度与生成缩放
  - 刷新分布与非重叠
  - 碰撞与移动/防卡死
  - 子弹战斗结算
  - 道具生效与过期
  - 胜负状态切换
- 交付要求：
  - 开发者必须先执行测试并确认全通过再交付。

## 10. 当前验证基线
- 现有测试数量目标已满足（>30）。
- 当前套件预期通过命令：
  - `go test ./...`
