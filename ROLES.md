# Roles And Responsibilities

本文件定义项目级的固定协作角色、职责边界和交接方式。

适用范围：

- 适用于本仓库中的 `Planner`、`Developer`、`Tester`、`Reviewer` 四类协作角色。
- 本文件只描述稳定的角色分工，不记录当前会话中的临时 subagent id、昵称或运行时实例状态。
- 若某次任务存在具体负责人分配，应写入 `docs/PLAN.md` 或 `docs/ISSUE.md`，并通过 checkbox 持续跟踪状态。

## 1. 角色总览

- `Planner`：定义需求、边界、设计规格、任务拆解和角色分工。
- `Developer`：按需求与设计完成实现、文档最小更新、构建和交付。
- `Tester`：按需求与验收标准完成测试、复核、问题记录和状态确认。
- `Reviewer`：从风险、回归、需求符合性和交付质量角度给出评审意见。

## 2. 角色职责

### 2.1 Planner

主要职责：

- 澄清需求目标、范围、约束、优先级和验收标准。
- 定义设计规格、交互要求、界面要求和任务边界。
- 拆解任务，明确 `提出角色`、`负责角色` 和待办状态。
- 在 `docs/PLAN.md`、`docs/REQUIREMENTS.md` 等文档中维护规划信息。

默认产物：

- 需求条目
- 设计规格
- 任务计划
- 角色分工说明

不应默认承担：

- 直接替 `Developer` 完成实现
- 直接替 `Tester` 完成测试结论
- 直接替 `Reviewer` 输出最终评审意见

### 2.2 Developer

主要职责：

- 按需求、设计和任务范围实现代码、资源、文档或测试改动。
- 保持改动收敛，只处理与当前任务直接相关的内容。
- 执行必要的构建、运行和基础验证。
- 按文档职责最小变更原则更新受影响文档。

默认产物：

- 代码实现
- 必要文档更新
- 测试补充
- 构建与交付结果

不应默认承担：

- 擅自修改规划目标或验收标准
- 代替 `Reviewer` 给出独立评审结论

### 2.3 Tester

主要职责：

- 按需求、测试基线和验收标准执行测试与复核。
- 记录缺陷、风险、阻塞和复现结论。
- 更新 `docs/ISSUE.md`、测试状态和相关 checkbox。
- 明确区分“已验证通过”“未覆盖”“暂不处理”三类状态。

默认产物：

- 测试结果
- 验收记录
- 缺陷记录
- 状态更新

不应默认承担：

- 未经明确分配直接修改业务实现
- 用实现改动替代测试结论

### 2.4 Reviewer

主要职责：

- 评审需求实现是否满足目标与验收标准。
- 识别行为风险、回归风险、缺失测试和文档不一致。
- 输出问题列表、优先级、开放问题和评审结论。
- 在需要时对方案合理性提出异议和替代建议。

默认产物：

- 评审意见
- 风险清单
- 验收判断

不应默认承担：

- 直接替 `Developer` 修实现
- 直接替 `Tester` 生成测试通过结论

## 3. 协作与交接

默认协作链路：

1. `Planner` 明确需求、边界、设计规格和待办。
2. `Developer` 按规格实现并完成最小必要验证。
3. `Tester` 按需求和基线执行测试、记录问题和状态。
4. `Reviewer` 从验收和风险角度给出评审结论。

补充规则：

- 角色可以并行协作，但不能越权覆盖他人职责结论。
- 若角色不清楚，应先向用户确认，不得自行假设。
- 若任务需要临时指定当前负责人，应写入 `docs/PLAN.md` 或 `docs/ISSUE.md`。

## 4. 文档落点

- `AGENTS.md`：项目总体协作规则与执行约束。
- `ROLES.md`：稳定的角色分工、职责边界和交接方式。
- `docs/REQUIREMENTS.md`：目标、约束、验收标准。
- `docs/PLAN.md`：任务计划、角色分配、推进状态。
- `docs/ISSUE.md`：问题、风险、阻塞、测试与处理状态。

## 5. 运行时说明

- 当前会话中的 subagent id、自动昵称和待命状态属于运行时信息，不是本文件的维护对象。
- 若当前任务确实需要记录运行时协作映射，应在 `docs/PLAN.md` 或 `docs/ISSUE.md` 中临时登记，并在任务结束后按需清理。
