# TankBattle Design

本文档描述当前代码实现已经落地的设计边界、模块职责和关键取舍，重点回答“为什么这样设计”以及“后续改动应该落在哪一层”。用户操作、运行命令和测试 API 细节分别以 `README.md`、`TEST-DESIGN.md`、`TEST-API.md` 为准，这里不重复维护说明性内容。

## 1. 设计目标与边界

- 目标是保持一个可直接运行的 Ebiten 桌面游戏主循环，同时让玩法逻辑、界面表现、调试能力和测试入口彼此解耦。
- 当前代码以“`internal/tankbattle` 维持单个运行时包，`cmd/` 只保留薄入口”的方式组织，还没有进一步拆成多个运行时子包。
- 本文档只描述已经存在的实现；如果未来进行目录重构或包拆分，需要在代码落地后再同步更新本文档。

## 2. 当前代码组织

当前仓库的运行时代码集中在 `internal/tankbattle`，按职责拆文件而不是拆包：

- `internal/tankbattle/main.go`
  - 暴露 `Run` 和 `RunWithOptions`，负责 Ebiten 窗口初始化、调试模式开关和顶层依赖装配。
- `internal/tankbattle/model.go`
  - 定义游戏常量、实体结构、枚举和聚合根 `game`。
- `internal/tankbattle/setup.go`
  - 负责新对局初始化、地图与堡垒构造、障碍布局和敌军生成。
- `internal/tankbattle/state.go`
  - 负责 `Update` 主时序、全局热键、暂停、历史面板、胜负判定和波次推进。
- `internal/tankbattle/menu.go`
  - 负责菜单输入和配置变更策略。
- `internal/tankbattle/movement.go`
  - 负责玩家移动、双击转向、敌军移动与 AI 决策。
- `internal/tankbattle/combat.go`
  - 负责子弹推进、碰撞、伤害结算和击毁反馈。
- `internal/tankbattle/powerup.go`
  - 负责道具刷新、拾取、生效和过期。
- `internal/tankbattle/render.go`
  - 负责战场、菜单、HUD、历史面板、消息框、暂停框和结算框的具体排版绘制。
- `internal/tankbattle/render_theme.go`
  - 负责主题色板、面板、描边、辉光、胶囊标签和进度条等通用 UI 原语。
- `internal/tankbattle/audio_manager.go` + `internal/tankbattle/audio_assets.go`
  - 负责音效资源、播放节流、开关与音量同步。
- `internal/tankbattle/settings.go` + `internal/tankbattle/score_history.go`
  - 负责用户配置和历史战绩的持久化。
- `internal/tankbattle/debug_api.go`
  - 负责测试调试控制器、本地 HTTP 接口、主循环内动作执行和快照导出。
- `internal/tankbattle/runtime_icon.go`
  - 负责 Windows 窗口图标加载。

命令入口位于：

- `cmd/tankbattle/main.go`
  - 仅调用 `tankbattle/internal/tankbattle.Run()`，不承载业务逻辑。
- `cmd/sfxgen/main.go`
  - 用于音效资源生成，不参与游戏运行时状态机。

测试资产集中在 `testing/`，设计/需求/测试文档集中在 `docs/`，运行期临时产物集中在 `.tmp/testing/`，不属于运行时设计的一部分。

## 3. 运行时装配

顶层运行从 `RunWithOptions` 开始，设计要点如下：

- 默认模式会加载用户配置、持久化历史数据并启用音频管理器。
- 当设置 `TANKBATTLE_DEBUG_API_ADDR` 时，进入调试模式：
  - 启用本地调试 API。
  - 禁用用户配置读取和本地数据写回。
  - 关闭音频依赖。
  - 固定随机种子为 `20260328`。
  - 允许窗口失焦时继续运行，便于功能和界面测试。

这样做的目的不是给游戏加第二套测试逻辑，而是在同一套运行时里切换“用户模式”和“可回归测试模式”。

## 4. 聚合根与状态模型

`game` 是当前实现的唯一聚合根，统一持有：

- 核心状态：`state`、`paused`、`win`、`frame`、`audioFrame`
- 实体集合：玩家、敌人、子弹、墙体、堡垒、爆炸、道具
- 对局数据：分数、击杀数、波次、消息、护盾/连发 buff
- 菜单配置：难度、总波数、音效开关、音量、菜单索引
- 历史数据：排行榜、滚动位置、显示状态
- 菜单恢复控制：`menuResumeAvailable`、`menuReturnState`、`menuReturnPaused`、`menuRequireRestart`
- 调试状态：`debug`、`debugFreeze`

主状态只有三个：

- `stateMenu`
- `statePlaying`
- `stateEnded`

当前没有再细分更多顶层状态，而是把“暂停”“历史面板”“消息框”设计成叠加状态或表现层开关。这样可以避免状态爆炸，同时让菜单恢复、暂停恢复和历史面板开关复用同一套对局数据。

## 5. 菜单与状态流转策略

菜单不是独立子系统，而是当前状态机的一个主状态入口。关键流转如下：

- 菜单开始游戏：`menuStart -> startMatch() -> statePlaying`
- 对局中进菜单：`enterMenuForConfig() -> stateMenu`
- 菜单返回原对局：`leaveMenuByToggle()`
- 对局结束：`finishMatch(win) -> stateEnded`
- 结算后回菜单：按 `M` 再次进入菜单

菜单恢复策略是当前实现里的一个关键设计点：

- 只改音效开关或音量时，菜单退出后恢复原对局。
- 改难度或总波数时，菜单退出后强制重开新对局。

这个策略通过 `menuResumeAvailable`、`menuReturnState`、`menuReturnPaused` 和 `menuRequireRestart` 集中表达，避免把“恢复旧局”或“重开新局”的判断散落到渲染层或输入层。

## 6. 主循环更新顺序

`Update()` 的时序遵循以下顺序：

1. 处理调试请求和调试冻结逻辑
2. 处理历史面板开关与自动收起
3. 处理重开和菜单切换
4. 按主状态分发菜单更新或结算态热键
5. 处理暂停切换
6. 推进帧计数、消息计时和 buff 计时
7. 更新玩家
8. 更新敌军
9. 更新子弹
10. 更新道具
11. 更新爆炸与墙体清理
12. 处理随机道具生成
13. 处理胜负判定和波次推进

这个顺序保证几件事：

- 输入优先于战斗推进，行为更可预测。
- 同一帧内尽量完成主要战斗结算，减少“看起来慢一拍”的状态延迟。
- 波次刷新和胜负判断基于已经更新后的最新场景数据。

## 7. 对局初始化与玩法子系统

`setup.go` 负责把一个新对局恢复到完整可运行状态，主要包含：

- 基于当前难度和波数设置对局参数。
- 重建堡垒、玩家、墙体和障碍。
- 选择并扰动障碍模板，保证布局不会堵死玩家或堡垒。
- 生成首波敌军，并同步战斗消息。

玩法相关的取舍如下：

- `movement.go` 专注“如何移动”和“何时转向”，包括玩家双击转向与敌军寻路节奏。
- `combat.go` 专注伤害来源、碰撞关系和击毁后果。
- `powerup.go` 专注 buff 生命周期，不跨层处理 HUD 和菜单。
- `state.go` 只负责时序和状态切换，不直接承担碰撞或寻路细节。

这种拆分方式的目的是让“规则变化”和“时序变化”分开演进，降低回归面。

## 8. 渲染架构

当前渲染层采用“两层结构”：

- `render_theme.go`
  - 提供主题色板和低层 UI 原语，例如面板、描边、角标、辉光、胶囊标签、进度条。
- `render.go`
  - 组合这些原语，完成菜单、HUD、历史面板、消息框、暂停框、胜负框以及战场对象的最终排版。

渲染顺序遵循：

1. 背景
2. 地图与静态对象
3. 玩家、敌军、子弹、道具、爆炸
4. HUD
5. 历史面板
6. 消息框 / 暂停框 / 结算框

关键设计点：

- 菜单场景是完整替换式绘制，不复用战斗层对象。
- 战斗场景中的 HUD 先于历史面板绘制，历史面板显示时再覆盖其上。
- 消息框、暂停框和结算框是最高层覆盖物，但历史面板打开时优先让历史面板独占上层阅读空间。
- `render.go` 中保留了较多固定布局常量，这是当前实现为了稳定对齐和快照回归做出的取舍。

## 9. UI 设计约束

最近一轮界面重做后，UI 采用统一的“战术控制台”视觉语言，当前代码中的约束是：

- 主菜单、HUD、历史面板和状态弹层共用同一套主题色与边框语义。
- 菜单的值列、胶囊标签和进度条使用固定列宽，避免随文案长度漂移。
- HUD 左右双栏布局固定，右侧状态框与威胁区避免重叠。
- 历史面板与菜单/HUD 共享面板体系，但独立占据中层到上层空间，保证可读性。
- 边框、角标和高光线已收细，避免视觉过重。

这些约束并不是通用 UI 规则，而是当前项目在 Ebiten 即时绘制模型下，为了兼顾美观、稳定布局和快照可回归性而做出的工程化选择。

## 10. 音频与持久化边界

音频和持久化都通过边界收口，避免污染核心逻辑：

- 音频通过 `sfxPlayer` 接口注入，`game` 只依赖播放、开关和音量能力。
- 默认运行时注入真实 `audioManager`，测试或调试模式可以传入 `nil` 或 mock。
- 用户设置和历史战绩分别由 `settings.go`、`score_history.go` 负责。
- 调试模式下禁止读写用户本地数据，避免测试污染真实环境。

这使得运行时主逻辑不需要关心文件格式、音频设备或测试替身的具体实现。

## 11. 调试 API 与可测试性设计

调试 API 的目标不是提供外挂控制，而是给功能测试和界面测试提供稳定驱动入口。当前实现采用：

- `RunWithOptions` 中根据环境变量启用本地 HTTP 服务。
- HTTP handler 不直接改游戏状态，而是把请求投递到 `DebugController.requests`。
- `Update()` 在主循环中消费这些请求，保证动作执行和快照导出仍发生在游戏线程语义内。
- 快照通过 `Draw + ReadPixels` 导出，不依赖操作系统截图。
- `scene.*` 动作用于构造命名界面场景，并通过 `debugFreeze` 冻结帧推进，保证 golden 快照稳定。

接口字段、动作名、调用示例和测试命令在 `TEST-API.md` 中维护；测试分层、目录结构和回归流程在 `TEST-DESIGN.md` 中维护。`DESIGN.md` 只保留这种设计为什么存在以及它与主循环的关系。

## 12. 目录与模块演进原则

当前代码已经从仓库根目录收口到 `internal/tankbattle`，但运行时仍保持单包，这和更彻底的模块化目标之间仍然存在张力。现阶段的约束是：

- 优先保持单聚合根和清晰的职责文件分工。
- 避免为了“看起来更整洁”而过早拆成大量小包，引入循环依赖和跨包噪声。
- 如果后续 `internal/tankbattle` 继续膨胀，优先考虑把稳定边界拆成更小的运行时子包，而不是先按函数机械拆散。

也就是说，当前设计承认目录还有进一步收敛空间，但文档必须首先反映“已经存在的结构”，而不是提前记录尚未实施的重构方案。

## 13. 文档职责边界

- `README.md`
  - 面向使用者，负责运行方式、操作说明和用户可见行为。
- `REQUIREMENTS.md`
  - 负责目标、约束和验收标准。
- `TEST-DESIGN.md`
  - 负责测试分层、流程和回归策略。
- `TEST-API.md`
  - 负责调试 API 契约和调用示例。
- `DESIGN.md`
  - 只负责解释当前代码设计、模块边界和架构取舍。

## 14. 何时更新本文档

出现以下变化时，应同步更新 `DESIGN.md`：

- 运行时代码从根包拆分到新的稳定包边界。
- 主状态机新增或删除顶层状态。
- 菜单恢复/重开策略发生变化。
- 调试模式对运行时装配的影响发生变化。
- 渲染分层或 UI 主主题体系发生明显变化。
- 音频、持久化、测试驱动边界发生重新划分。
