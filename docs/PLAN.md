# 界面重设计实施计划

## 1. 需求分解

目标：在不修改核心玩法、操作方式与数值规则的前提下，重新设计游戏界面，让整体观感更精致、信息层级更清晰、战斗反馈更有质感。

本次改造覆盖：

- 主菜单视觉重设计
- 对局 HUD 与状态面板重设计
- 战场背景、堡垒、墙体、坦克、子弹、爆炸、道具等视觉表现增强
- 历史战绩面板风格统一
- 与界面改造直接相关的测试与用户文档更新

本次不做：

- 不引入新的玩法规则、敌人类型或数值系统
- 不新增外部美术资源生成链路，优先使用代码绘制完成重设计
- 不改动输入映射与已有交互语义

## 2. 合理性分析与实现取舍

这次需求合理，且适合分层推进。

原因：

- 当前界面主要由纯色矩形和默认调试字体直接拼接，完成度够用，但风格统一度、视觉重心、反馈层次都偏弱。
- 现有渲染逻辑集中在 `render.go`，可以在不改玩法逻辑的前提下完成表现层重构，回归风险可控。
- 如果直接引入新的图片资产或复杂动画系统，交付成本和回归面会明显扩大；本次优先采用“代码绘制 + 小幅结构整理”的方案，收益更稳定。

备选但暂不采用的方案：

1. 外接整套图片 UI 资源
   - 成本：高
   - 风险：资源风格统一、缩放适配、透明边缘与集成回归成本高
   - 回退：可回退，但投入浪费较大

2. 只微调颜色和间距
   - 成本：低
   - 风险：低
   - 问题：提升有限，无法达到“重新设计界面、效果更好”的目标

最终采用：

- 以代码绘制为主，补充统一主题色、面板体系、选中态、层次阴影、动态脉冲和更清晰的信息布局。

## 3. 实施方案

### 阶段 A：建立统一视觉语言

目标：

- 提炼新的 UI 调色板与辅助绘制函数
- 统一面板边框、阴影、发光、强调线、文本强调风格
- 为菜单、HUD、历史面板、状态弹层提供复用能力

实现要点：

- 新增独立的 UI 主题/绘制辅助文件，降低 `render.go` 的表现层耦合
- 提供带高光、描边、角标、进度条、脉冲动画的基础绘制能力

### 阶段 B：重做主菜单

目标：

- 让菜单从“设置列表”升级为“任务控制台”风格
- 强化标题区、说明区、当前难度与快捷键信息的层次
- 提升选中项反馈与可读性

实现要点：

- 增加主视觉标题区、副标题和说明卡片
- 把设置项改为更明显的卡片式布局，加入选中高亮、数值条和状态标签
- 增加右侧信息区，集中展示快捷键、当前任务摘要、开始提示

### 阶段 C：升级对局 HUD 与状态面板

目标：

- 提升 HUD 的“指挥面板”质感
- 让波次、分数、敌人数、堡垒血量、坦克状态的层级更清晰
- 优化暂停、胜利、失败和消息提示的观感

实现要点：

- 重构顶部 HUD 卡片布局与条形信息表达
- 强化 buff 激活态、危险态、低血量警示
- 改造消息条、暂停框、结算框的视觉样式

### 阶段 D：增强战场视觉反馈

目标：

- 提升战场空间层次感与战斗冲击力
- 保持可读性优先，避免华而不实

实现要点：

- 重做背景分层、扫描线/网格/氛围光
- 优化砖墙、钢墙、堡垒、坦克、子弹、爆炸、道具的绘制细节
- 为关键对象增加轻量动态效果，如脉冲、尾迹、外圈辉光

### 阶段 E：统一历史面板与回归验证

目标：

- 让历史战绩面板与新菜单/HUD 使用同一视觉语言
- 补充或调整与布局相关的测试
- 更新必要文档并完成构建、运行验证

实现要点：

- 重绘历史面板的标题区、表头区、行样式和底部状态信息
- 更新 `render_test.go`
- 更新 `README.md`
- 视设计边界变化决定是否更新 `DESIGN.md`

## 5. 验收标准

- 菜单、HUD、历史面板、状态弹层整体风格统一，明显优于当前纯矩形拼接效果
- 主要信息区域无重叠、无遮挡、无明显对齐问题
- 战场主体与特效增强后，仍保持玩家、敌人、堡垒、子弹、道具的快速可辨识性
- 原有操作和玩法行为保持不变
- `go build ./...` 通过，GUI 可执行文件可成功构建并启动
