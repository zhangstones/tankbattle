# 背景简化方案计划

## 1. 目标

当前技术实现不适合继续追求复杂背景重设计。
本轮目标调整为：

- 用低风险、低复杂度方案，把背景稳定收敛到“地面坦克战场”语义。
- 背景只提供基础氛围，不承担复杂美术表达。
- 优先保证主体可读性、快照稳定性和实现成本可控。

## 2. 采用方案

采用“纯色背景 + 弱网格 + 少量沙石模拟”方案。
具体要求：

- 纯色地表底色为主，不再追求复杂地貌分层。
- 保留很弱的地表网格，作为战场秩序感来源。
- 只允许少量沙石、尘土点或轻微地表噪声作为点缀。
- 禁止使用大面积强辉光、漂浮亮点、强扫描线、雷达感线条。
- 禁止继续堆叠复杂战壕、防线、推进通路和大构图装饰。

## 3. 设计取舍

放弃复杂方案的原因：

- 当前实现手段更适合稳定的程序化简背景，不适合高完成度美术场景。
- 复杂背景容易把主体压暗、压脏，反而削弱战斗可读性。
- 复杂构图会显著提高 golden 维护成本和界面回归风险。

保留简化方案的收益：

- 题材识别仍然正确。
- 视觉更干净，HUD、坦克、子弹和堡垒更容易看清。
- 实现简单、风险低、易于长期维护。

# 技能修订计划

## 1. 目标

本轮目标不是新增技能，而是系统修订现有技能文档，补齐明显缺失、纠正误导性步骤，并让技能描述更泛化、更通用、更容易在相似仓库中复用。

重点要求：

- 每个技能都要补齐适用场景、边界、输入输出、失败回退和完成标准。
- 修正与仓库协作规则或代理执行约束冲突的内容。
- 对仓库特有路径、脚本和命令，明确区分“默认实现”与“可替换示例”。
- 避免把技能写成只适用于当前一次任务的操作备忘录。

## 3. 每个技能的统一修订要求

- 明确适用场景与不适用场景，避免误触发。
- 明确开始前应确认的输入信息。
- 给出稳定的执行流程，而不是只列原则。
- 补齐失败处理、例外条件和回退路径。
- 明确交付产物、验证方式和完成标准。
- 删除过度绑定当前仓库、当前机器或单次任务的表达；若必须引用仓库默认路径，需说明其替换边界。

## 4. 交付方式

- 逐个技能修订并核对内容。
- 全部剩余技能修订完成后统一提交。
- 提交说明需注明：
  - 本轮修订了哪些技能
  - 修正了哪些误导性规则或缺失项
  - 为什么不需要更新 README 之外的其他项目文档

## 5. 完成标准

- 剩余技能都已按统一标准修订完成。
- 技能之间不存在明显职责冲突或重复指令。
- 技能内容不再与当前仓库 AGENTS 规则或代理执行约束冲突。
- 技能既能指导当前仓库，也能在相似项目中复用。
