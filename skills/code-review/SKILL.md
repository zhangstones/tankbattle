# 代码审查技能

## 审查目标

- 优先发现会导致错误结果、崩溃、数据损坏、性能退化或安全风险的问题。
- 识别需求与实现不一致、测试覆盖缺失、边界条件遗漏。
- 在保证准确性的前提下给出最小可执行修复建议。

## 审查流程

1. 明确审查范围：分支差异、目标文件、需求上下文。
2. 先看高风险路径：状态机、并发、I/O、边界值、错误处理。
3. 再看行为回归：对比变更前后输入输出与副作用。
4. 核对测试：已有测试是否覆盖关键路径，是否缺功能测试/回归测试。
5. 最后看风格与设计：命名、重复逻辑、模块边界、耦合度。

## 严重级别定义

- `S0` 阻断：会导致功能不可用、数据错误或明显崩溃。
- `S1` 高：高概率线上问题或显著行为偏差。
- `S2` 中：潜在缺陷、边界条件风险、维护成本明显增加。
- `S3` 低：风格一致性、可读性或小幅重构建议。

## 输出格式要求

1. 先列 Findings，按 `S0 -> S1 -> S2 -> S3` 排序。
2. 每条 Finding 必须包含：
   - 问题描述（是什么）
   - 影响范围（为什么重要）
   - 文件定位（绝对路径 + 行号）
   - 建议修复（怎么改）
3. 若无问题，明确写“未发现阻断或高风险问题”，并说明残余风险与测试盲区。
4. 总结放在 Findings 后，不可置前。

## 快速检查清单

- 需求是否被完整实现，是否存在隐藏假设。
- 错误处理是否丢失，返回值是否被忽略。
- 是否引入新状态但未初始化/重置。
- 是否存在重复逻辑导致未来不一致。
- 测试是否覆盖：正常路径、边界路径、失败路径。
- 文档是否与行为一致（README/需求说明/配置说明）。

## 审查后动作

- 对 `S0/S1` 问题优先给出可直接落地的补丁方案。
- 必要时补充最小回归测试，防止问题复发。
- 结论中区分“已验证”与“待验证”项。
