# 代码审查技能

## 适用场景

- 用户明确要求 `review`、`代码审查`、`检查问题`、`找风险`。
- 需要评审一个 PR、提交差异、工作区未提交改动，或某几个指定文件。
- 需要在“不直接改代码”的前提下，先判断实现是否存在缺陷、回归或遗漏。

## 不适用场景

- 用户明确要求直接实现、修复、重构，而不是先做审查。
- 任务目标是解释代码、总结架构、写文档，而不是判断问题严重性。
- 没有可审查对象，只有抽象方案讨论。

若用户同时要求“先 review 再修”，先完成审查结论，再进入修复。

## 审查目标

- 优先发现会导致错误结果、崩溃、数据损坏、安全风险、明显性能退化或用户可见行为错误的问题。
- 识别需求与实现不一致、测试覆盖缺失、边界条件遗漏、状态流转不完整。
- 在保证准确性的前提下给出最小可执行修复建议，而不是泛泛而谈。

## 审查输入

开始前先尽量明确以下输入；若用户未给全，按当前上下文补齐：

- 审查范围：PR、commit、分支差异、工作区改动、指定文件。
- 需求上下文：需求文档、Issue、用户描述、已有行为。
- 验证线索：相关测试、构建命令、复现步骤、日志。

若范围不明确，优先按“用户刚提到的改动”或“当前未提交差异”审查，不自行扩大到整个仓库。

## 审查流程

1. 明确审查范围：变更集、目标文件、需求上下文、预期行为。
2. 先看高风险路径：状态机、并发、I/O、边界值、错误处理、资源释放、权限与鉴权。
3. 再看行为回归：对比变更前后输入输出、副作用、持久化、配置兼容性和用户可见表现。
4. 核对测试：已有测试是否覆盖关键路径，是否缺功能测试、回归测试或失败路径测试。
5. 最后看设计与维护性：重复逻辑、模块边界、隐式耦合、未来容易出错的实现方式。

## 证据标准

- 每条结论都应基于可定位证据：代码路径、行号、调用链、测试缺口或明确的行为推演。
- 不要把“编码风格偏好”包装成缺陷。
- 若结论依赖假设，必须明确写出假设条件。
- 若无法确认是否为缺陷，降级为“风险/疑问”，不要伪装成确定性结论。

## 严重级别定义

- `S0` 阻断：会导致功能不可用、数据错误、严重安全问题或明显崩溃。
- `S1` 高：高概率线上问题、显著行为偏差、重要路径回归。
- `S2` 中：潜在缺陷、边界条件风险、兼容性问题、维护成本明显增加。
- `S3` 低：可读性、风格一致性、轻度设计改进建议。

默认只把真正影响正确性、稳定性、回归风险的问题放进 Findings；纯偏好建议可放在补充说明。

## 输出格式要求

1. 先列 Findings，按 `S0 -> S1 -> S2 -> S3` 排序。
2. 每条 Finding 必须包含：
   - 问题描述：是什么问题
   - 影响范围：为什么重要，会影响什么行为
   - 文件定位：绝对路径 + 行号
   - 触发条件：在什么输入、状态或场景下出现
   - 建议修复：怎么改最稳妥
3. 若无问题，明确写“未发现阻断或高风险问题”，并补充残余风险、测试盲区或未验证项。
4. 总结放在 Findings 后，不可置前。
5. 若用户要求简短输出，也不能省略文件定位和影响说明。

## 快速检查维度

- 需求是否被完整实现，是否存在隐藏假设或遗漏状态。
- 错误处理是否丢失，返回值、异常或失败分支是否被忽略。
- 是否引入新状态但未初始化、未重置或未持久化。
- 是否存在重复逻辑、分叉实现或兼容旧行为时的不一致。
- 测试是否覆盖正常路径、边界路径、失败路径和回归路径。
- 文档、配置说明、接口契约是否与行为一致。
- 删除或重命名字段时，是否处理了兼容性和迁移成本。

## 审查深度控制

- 小改动：重点看直接修改的行为和相邻调用链，不做过度扩散。
- 中等改动：补看相关状态流、配置、测试和用户可见输出。
- 高风险改动：追到入口、核心状态、持久化、副作用边界和回归验证链路。

不要为了“显得全面”而把整个仓库都扫一遍；审查深度应与变更风险匹配。

## 审查后动作

- 对 `S0/S1` 问题优先给出可直接落地的补丁思路；若用户要求，可继续实施修复。
- 必要时建议补最小回归测试，避免问题复发。
- 结论中区分“已验证”“代码推断”“待验证”三类依据。
- 若仓库存在未读文档或未运行测试导致的不确定性，要明确写出，而不是装作已确认。
