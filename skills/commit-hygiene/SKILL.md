# 提交卫生

## 适用场景

- 用户要求整理提交、准备提交、拆分改动、统一提交信息。
- 一个任务已经完成，需要把代码、文档、测试整理成可审查提交。
- 当前工作区改动较多，需要判断哪些改动属于同一问题域。

## 不适用场景

- 用户明确要求暂不提交。
- 当前任务还没完成，验证还没结束，提交边界还不稳定。
- 需要改代码解决问题时，先完成实现和验证，再整理提交。

## 核心目标

- 一个提交只表达一个清晰的问题域或结果。
- 提交标题能直接说明改动目的，而不是罗列文件。
- 不为了“历史好看”而重写用户未授权的历史。

## 开始前确认

- 当前任务范围是什么，哪些文件属于本次任务。
- 工作区里是否存在其他人或之前任务遗留的无关改动。
- 用户是否明确要求 `amend`、`squash`、`rebase` 或改写历史。

## 提交粒度规则

- 同一问题域的代码、测试、职责匹配文档，应尽量放在同一提交内。
- 不同问题域分开提交，例如“布局修复”和“音频行为修复”不要混在一起。
- 大型任务可拆阶段提交，但每个阶段都必须可解释、可验证、可回退。
- 不要把顺手格式化、无关重命名或清理改动混入当前提交。

## 历史整理边界

- 默认新增普通提交，不主动改写历史。
- 只有在用户明确要求，或当前提交尚未共享且确实需要整理时，才考虑 `--amend` 或 squash。
- 不使用交互式 git 控制台；若必须整理历史，优先使用可审计的非交互命令。
- 未经明确要求，不执行会重写历史的 `reset --soft`、`rebase`、`commit --amend`。

## 提交信息规则

- 标题格式：`<prefix>: <一句话结果>`
- 常用前缀：
  - `fix:` 修复缺陷、回归、错误行为
  - `feat:` 新功能、新交互、新配置
  - `refactor:` 不改行为的结构调整
  - `docs:` 纯文档更新
  - `test:` 测试补充或调整
  - `chore:` 辅助性维护，不直接影响功能
- 标题应描述结果或目的，不写成文件清单。

示例：

```text
fix: stabilize menu resume flow
```

```text
docs: clarify UI testing workflow
```

## 标准流程

1. 先看 `git status --short`，确认本次任务相关文件。
2. 排除无关改动，不要误加到当前提交。
3. 按问题域分组暂存。
4. 在提交前完成本任务要求的验证。
5. 用清晰标题提交。
6. 提交后再检查一次 `git status --short` 与 `git log --oneline -n 5`。

## 常见错误

- 把多个无关修复塞进一个提交。
- 提交标题描述“改了什么文件”，而不是“解决了什么问题”。
- 在存在未验证改动时提前提交。
- 未经用户授权改写历史。
- 为了整洁把其他人的进行中改动一起整理掉。

## 完成标准

- 当前提交范围与任务范围一致。
- 提交信息能独立解释本次改动。
- 必要验证已完成，或已明确记录未验证原因。
- 工作区只剩未提交的有意保留改动，不存在误带文件。
