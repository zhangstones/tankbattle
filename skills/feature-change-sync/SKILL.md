# 功能变更联动同步

## 执行流程

1. 先用一句话确认本次功能范围，并列出受影响模块/文件。
2. 优先更新 `REQUIREMENTS.md`，补齐行为、约束与验收点。
3. 更新 `README.md`，反映用户可见变化（操作、配置、限制、启动方式等）。
4. 新增或更新功能测试，证明端到端行为正确。
5. 当出现以下任一情况时，功能测试案例为强制项，不得省略：
- 修改代码逻辑（包括规则、流程、判定、状态切换、输入处理等）
- 新增用户可见特性或可配置项
- 影响已有行为的重构或性能优化
6. 仅在功能测试难以覆盖的逻辑分支上补充单元测试。
7. 执行完整测试命令（`go test ./...`）并修复回归。

## 完成清单

- `REQUIREMENTS.md` 已反映最新功能契约。
- `README.md` 已用用户视角描述新行为。
- 功能测试至少覆盖正常路径与一个边界/失败路径。
- 测试结果为通过。

## 约束规则

- 仅改代码、不改文档与测试，不得标记为完成。
- 修改代码逻辑或新增特性时，必须新增/更新对应功能测试案例。
- 同一功能的文档/测试不得延期到后续提交。
- 若本次行为刻意不变，必须在任务说明中明确写出。
