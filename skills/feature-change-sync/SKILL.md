# 功能变更联动

## 适用场景

- 新增、修改、删除用户可见功能。
- 修复 Bug，导致已有行为发生变化。
- 修改规则、流程、状态切换、输入处理、配置项或接口契约。

## 不适用场景

- 纯文档润色，不影响功能契约。
- 纯重命名、格式化或等价重构，且行为、接口、测试入口都不变。
- 一次性调试日志或临时排查代码，且不会进入正式交付。

若“看起来只是重构”，但已经影响用户可见行为、接口、配置或测试入口，仍按功能变更处理。

## 核心目标

- 代码变更、测试覆盖和文档说明保持一致。
- 文档按职责最小更新，不重复维护同一信息。
- 不把“后续再补文档/测试”当成默认做法。

## 开始前确认

- 本次改动到底改变了什么行为。
- 这项变化是用户可见、开发者可见，还是仅内部结构变化。
- 哪些文档职责会被影响。
- 现有测试是否已经覆盖，还是必须新增/调整。

## 文档职责边界

- `README.md`：用户如何运行、使用、观察到什么行为。
- 需求文档：目标、约束、验收标准、行为契约。
- 设计文档：架构边界、设计取舍、实现方案。
- FAQ / 测试专项文档：高频问题、排障、测试入口、测试矩阵。

规则：

- 只更新职责匹配的文档。
- 若某类文档不受影响，不要机械同步；但应在任务说明或提交说明中写明原因。

## 测试规则

- 只要代码逻辑有变化，就默认需要测试更新。
- 功能变更优先补功能测试或回归测试，再考虑单元测试。
- 修复 Bug 时，优先补能稳定复现问题的最小回归测试。
- 只有在功能测试不适合覆盖时，才把验证责任主要放在单元测试。

## 标准流程

1. 用一句话写清本次功能变化。
2. 列出受影响的代码、测试和文档。
3. 先改代码，再补或调整测试。
4. 按职责更新文档，而不是所有文档一起改。
5. 运行本次任务要求的验证命令。
6. 交付时说明哪些文档更新了，哪些没更新以及原因。

## 常见遗漏

- 改了行为但没有补测试。
- README 写了新行为，但需求文档没有体现约束变化。
- 设计文档写了很多实现细节，却漏了真正影响用户的说明。
- 修了代码却没同步测试场景、配置说明或调试入口。

## 完成标准

- 行为变化已有对应测试覆盖，或明确说明为何无法自动化覆盖。
- 职责匹配的文档已同步更新。
- 未更新的文档类型已说明原因。
- 交付时不存在“代码已变，但文档/测试仍描述旧行为”的状态。
