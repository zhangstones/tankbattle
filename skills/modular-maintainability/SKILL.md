# 模块化可维护性

## 模块化规则

1. 按职责拆分，不按改动批次拆分：初始化、状态、渲染、移动、战斗、菜单分别负责。
2. 模块接口保持小而明确，避免跨模块隐式写状态。
3. 优先通过参数传递依赖，减少全局可变状态。
4. 尽量保持单向数据流，例如“状态更新 -> 快照/视图模型 -> 渲染”。
5. 文件职责必须聚焦，出现多领域混杂时及时抽离。

## 渐进式重构顺序

面对高风险重构时，优先使用“先收边界，再搬实现”的顺序：

1. 先收根目录或外围入口，保留对外稳定路径。
2. 先抽独立基础模块：音频、存储、调试 API、配置读写等。
3. 涉及 UI 时，先定义稳定快照或 ViewModel，再移动渲染代码。
4. 最后再搬核心运行时或玩法逻辑，不要在同一步里顺手改业务语义。

不要把“目录重组”“包拆分”“玩法重写”“视觉改版”揉成一次提交。

## 稳定边界策略

1. 先找到天然边界，再决定目录边界。
2. 对外保留薄封装入口，避免一次性改穿所有调用方。
3. UI 不直接吃核心聚合根时，优先引入 `Snapshot` / `ViewModel` 过渡层。
4. 副作用能力先抽到独立包，运行时只保留桥接逻辑。
5. 若某层仍大量读写另一层内部字段，说明边界还没稳定，不要急着继续拆。

## 触发重构阈值

满足任一条件应在合并前重构：

- 单函数超过约 60-80 行且混合多种职责。
- 同一文件频繁因无关原因被修改。
- 一个新功能需要改动多个无关模块才能落地。
- 编写测试必须依赖大量内部 mock 才能验证。

## 可测试性要求

- 每个模块都应暴露可独立验证的行为。
- 路径、碰撞、规则判定优先做成确定性纯函数。
- I/O、UI、时序等副作用应放在边界层并收敛为窄接口。
- 重构 UI 时，优先让快照测试直接消费稳定输入，而不是依赖桌面截图。
- 重构核心运行时时，保留一条端到端验证链路，确认入口行为没有被目录迁移破坏。

## 每阶段验证规则

每完成一个重构阶段，都单独做以下动作：

1. `gofmt`
2. `go test ./...`
3. `go build ./...`
4. 若有 GUI 交付，构建 GUI 可执行文件并实际启动验证。
5. 若项目有真实 E2E 或快照套件，单独跑一次，不要只依赖单元测试。

只在这一阶段验证通过后再进入下一阶段。

## 提交粒度规则

1. 每个稳定阶段单独提交，保证可回退。
2. “抽基础模块”“抽 UI”“搬核心运行时”至少分成独立提交。
3. 文档与代码同步收口，但只更新职责匹配的文档。
4. 若发现原计划顺序风险过高，先调整顺序，再实施，不要硬拆。

## 变更完成标准

- 修改后模块边界仍清晰。
- 新增代码在对应模块测试文件中有匹配测试。
- 不保留会增加耦合的临时性捷径。
- 每一步迁移都有对应验证记录，并能解释为何这个顺序更稳。
