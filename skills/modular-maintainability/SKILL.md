# 模块化可维护性

## 模块化规则

1. 按职责拆分，不按改动批次拆分：初始化、状态、渲染、移动、战斗、菜单分别负责。
2. 模块接口保持小而明确，避免跨模块隐式写状态。
3. 优先通过参数传递依赖，减少全局可变状态。
4. 尽量保持单向数据流（例如：状态更新 -> 渲染）。
5. 文件职责必须聚焦，出现多领域混杂时及时抽离。

## 触发重构阈值

满足任一条件应在合并前重构：

- 单函数超过约 60-80 行且混合多种职责。
- 同一文件频繁因无关原因被修改。
- 一个新功能需要改动多个无关模块才能落地。
- 编写测试必须依赖大量内部 mock 才能验证。

## 可测试性要求

- 每个模块都应暴露可独立验证的行为。
- 路径、碰撞、规则判定优先做成确定性纯函数。
- I/O、UI、时序等副作用应放在边界层并收敛为窄接口。

## 变更完成标准

- 修改后模块边界仍清晰。
- 新增代码在对应模块测试文件中有匹配测试。
- 不保留会增加耦合的临时性捷径。
