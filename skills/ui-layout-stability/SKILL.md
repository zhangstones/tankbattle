# 界面布局稳定性

## 适用场景

- 菜单/HUD/状态框新增文案、选项或控件后出现拥挤、重叠、错位。
- 反复通过“微调像素”修问题，导致同一问题多次回归。
- 文案长度变化（中英文、大小写、说明合并）触发布局抖动。

## 核心原则

1. 先定义区域，再放元素：标题区、说明区、选项区、底部提示区必须有清晰边界。
2. 优先“计算布局”，不要堆硬编码坐标：通过可用高度/宽度推导间距和起点。
3. 文本对齐必须函数化：横向/纵向居中统一使用 `centeredTextX/Y`。
4. 说明文案按语义分组：同类操作放同一行（例如 `FIRE J/Space` 与 `Combat` 同行）。
5. 用测试锁几何关系，而不是锁具体像素值。

## 推荐实现模式（菜单）

1. 固定主容器尺寸与内边距（panel + inner panel）。
2. 设定标题区高度，标题文本使用“区域居中”计算。
3. 说明区使用固定行高，底部留最小缓冲。
4. 选项区起点 = 说明区底部 + 安全间距。
5. 选项行距根据“可用空间”计算并夹紧到合理区间（min/max）。
6. 底部提示区单独保留，不参与选项区抢占。

## 必备回归测试

1. 选项区底部 < 底部提示区顶部（不重叠）。
2. 说明区底部 < 选项区顶部（不挤压）。
3. 说明区与选项区间距在可读范围（如 8~20px）。
4. 标题在标题栏内水平居中。
5. 标题在标题栏内垂直居中。
6. 关键提示文案属于正确语义行（如 `FIRE J/Space` 在 Combat 行）。

## 变更流程

1. 先收敛布局参数到常量与计算函数。
2. 再调整文案与视觉样式，不直接改散落坐标。
3. 修改后立即补/改几何回归测试。
4. 运行 `go test ./...` 与 `go build ./...`。
5. GUI 目标需构建并启动验证。
