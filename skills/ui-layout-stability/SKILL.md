# UI 布局稳定性

## 适用场景

- 菜单、HUD、面板、弹层在多轮迭代后出现对齐漂移、层级混乱或信息拥挤。
- 新增信息项后，界面开始频繁重排、重叠或视觉失衡。
- 需要把一次 UI 调整沉淀为长期可复用的布局约束。

## 不适用场景

- 只是单个颜色、文案或局部样式微调，不影响布局。
- 任务是整体视觉主题重做，而不是稳定现有布局。
- 还没确认信息层级，就直接做像素级微调。

## 核心目标

- 保持信息分层稳定，不让高频信息位置反复漂移。
- 让对齐、间距和分区由统一规则驱动，而不是逐处补丁。
- 在新增内容时保持整体扫描路径和可读性。

## 开始前确认

- 哪些是核心高频信息，哪些是次要状态，哪些是操作提示。
- 当前问题属于重叠、错位、拥挤、跳区还是视觉主次失衡。
- 这次改动主要调整的是布局、样式还是文案。

## 设计原则

- 先分层，再分区，再调细节。
- 同类信息长期保持同一区域，避免频繁换位。
- label/value、标题/正文、主信息/辅助信息应有统一对齐规则。
- 主次通过层级和留白表达，不靠堆叠装饰。
- 一次只改一个维度：布局、样式、文案三者尽量不要同轮大改。

## 标准流程

1. 先确定信息分层和区域职责。
2. 固定主要锚点、列宽、行高或容器边界。
3. 调整单个问题，例如列对齐、面板间距或弹层避让。
4. 构建并观察关键状态画面，不只看单一截图。
5. 若布局已稳定，再决定是否需要同步样式微调。

## 典型检查点

- 核心信息是否处于高频视线区。
- 同类信息是否在不同状态下保持稳定位置。
- 文本是否出现超框、截断、错列或数值抖动。
- 面板、HUD、弹层之间是否有遮挡或压迫。
- 新增信息是否破坏原有扫描路径。

## 联动要求

- 若布局规则发生稳定性变化，应补对应布局约束测试或快照验证。
- 若用户可见交互提示、说明文字或信息结构变了，再更新职责匹配文档。
- 不要把纯视觉风格争议伪装成布局缺陷。

## 完成标准

- 核心信息层级清晰，扫描路径稳定。
- 同类信息位置不再随状态频繁跳变。
- 对齐、间距和避让规则可解释且可复现。
- 代码中没有遗留相互冲突的试验性布局分支。
